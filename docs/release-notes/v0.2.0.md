# v0.2.0 — Evaluator Refresh

**Release date:** 2026-04-14
**Scope:** `rag-forge-evaluator` (Python). `rag-forge-core`, `rag-forge-observability`, `@rag-forge/cli`, `@rag-forge/mcp`, and `@rag-forge/shared` bumped in lockstep for monorepo version alignment.

## Why this release exists

The cycle-2 clinical audit on v0.1.3 (2026-04-14) exercised RAG-Forge against a real clinical RAG pipeline and surfaced four structural problems with the evaluator:

1. **The RAGAS integration was version-fragile.** The adapter relied on `ragas.evaluate()` picking its own default LLM and embeddings, which changed between ragas 0.2.x and 0.4.x. On ragas 0.4.x it crashed with `AttributeError: 'OpenAIEmbeddings' object has no attribute 'embed_query'` and truncated long structured responses at `max_tokens`.
2. **Exceptions were silently coerced to score 0.0.** A broken ragas run reported `Overall: 0.0000, Scored: 48, Skipped: 0` — operators saw all zeros and concluded their RAG was broken, when the framework itself was broken.
3. **Safety refusals were penalized as quality failures.** When a pipeline correctly refused to answer because the knowledge base lacked sufficient support — exactly what safety-conscious RAGs in regulated industries are designed to do — the evaluator scored it as a non-answer and tanked the metrics.
4. **The audit report was unprofessional.** Text truncated, HTML bleeding off-screen, per-sample detail inaccessible in the PDF, alignment inconsistent. The single most customer-facing artifact RAG-Forge produces undermined the whole tool.

v0.2.0 addresses all four at the architectural root.

## ⚠️ Breaking changes

Read this section before upgrading. Two things shift by default.

### 1. Scores will shift on pipelines with safety refusals

Refusal-aware scoring is **default-on** in v0.2.0. If your pipeline has a strict "don't fabricate" guardrail — common in medical, legal, financial, and compliance RAGs — samples where it correctly refuses to answer will now score higher than they did in v0.1.3. This is a correction, not a regression: the old scores were wrong in a systematic way.

**To preserve v0.1.x scoring semantics** (e.g., for CI gates that were tuned to the old numbers), pass `--strict`:

```bash
rag-forge audit --golden-set qa.json --judge claude --strict
```

### 2. The `ragas` optional extra is now pinned to `>=0.4,<0.6`

Previous pin: `ragas>=0.2.10,<0.3`. If you have ragas 0.2.x pinned in your own environment, upgrade to 0.4.x before upgrading `rag-forge-evaluator`. Fresh installs handle this automatically.

### 3. `--evaluator ragas --judge claude` is now supported

It previously errored out with a `ConfigurationError`. If any CI script relied on that error as a signal, update it.

## What's new

### Refusal-aware scoring (default-on)

Before the judge scores each sample, it inline-classifies the response as `standard` or `safety_refusal`. Safety refusals are scored against a refusal rubric that rewards acknowledging the question and explaining why it can't be answered — not penalizing the non-answer. The classification happens **in the same judge call** as the scoring, so there's zero extra API cost.

A new `Safety Refusals` section in the audit report shows which samples were classified as refusals and the judge's one-line justification for each. If more than 30% of samples are classified as refusals, the report renders an amber warning banner asking operators to spot-check — the mode trusts the judge's classification, and that's the risk.

**Worked example:** a clinical RAG audit refused a parent's request for an exact paediatric metformin dose because the knowledge base contains adult dosing only. Under v0.1.3 this scored 0.47 on answer relevance and 0.24 on context relevance — a safety win misclassified as a quality failure. Under v0.2.0 the judge classifies it as `safety_refusal` with the justification *"Context lacks paediatric dosing; response correctly directs to clinical supervision without fabricating a dose."* and the scores reflect the refusal rubric instead of the standard one.

Opt-out with `--strict` or `--no-refusal-aware`.

### RAGAS adapter rebuilt around injected wrappers

New module `rag_forge_evaluator.engines.ragas_adapters` exposes two wrapper classes:

- **`RagForgeRagasLLM`** — implements the full ragas 0.4.x `BaseRagasLLM` contract (`generate_text` / `agenerate_text` with `n`, `temperature`, `stop`, `callbacks` parameters, returning `LLMResult`). Forwards all calls to rag-forge's existing `Judge` abstraction so `--judge claude` finally works end-to-end with ragas (the v0.1.2 promise, two versions late).
- **`RagForgeRagasEmbeddings`** — pluggable embeddings provider: `openai` (text-embedding-3-small), `voyage` (voyage-3 via the new `[ragas-voyage]` optional extra), or `mock` for tests.

`ragas_evaluator.py` now injects these wrappers explicitly instead of letting ragas pick version-fragile defaults. The result: ragas upgrades can't silently break the adapter because we no longer depend on ragas's default picker at all.

### Top-tier audit report

The HTML and PDF reports were rebuilt around a single Jinja template with a shared browser/print stylesheet. Fix once, both formats benefit.

New information architecture:

- Header strip with report number, date, cycle, UTC timestamp
- RMM hero callout with historical sparkline and plain-English level explanation
- Full six-rung maturity ladder with current level highlighted
- Executive summary prose
- **TL;DR callout** — two columns (what's working / what needs fixing) plus a priority next step
- Metric breakdown table with a plain-English "What it measures" column
- Cost summary block with token split and CI projection
- Safety Refusals section (new)
- Worst performing case callout
- Per-sample detail cards — collapsible in the browser, fully expanded in print
- Skipped samples section (only renders when non-empty)
- **Compliance & Attestation footer** — method disclosure, data handling, regulatory alignment references (NIST AI RMF 1.0 § 4.3, ISO/IEC 42001:2023 § 8.3, EU AI Act Article 15), attestation grid, limitations disclaimer

Playwright PDF generation now calls `page.emulate_media(media="print")` before rendering, so the template's `@media print` rules apply cleanly — no more text bleed, no more truncated per-sample detail, proper page breaks between sample cards.

### Skipped-sample accounting replaces silent zeros

`EvaluationResult` gains a `skipped_samples: list[SkipRecord]` field. When the RAGAS engine can't extract a metric score (parse failure, whole-batch crash, API error), the skip is recorded with sample ID, metric name, reason, and exception type — and surfaced in both the JSON and HTML/PDF reports. No more silent 0.0 coercion, no more operators mistaking broken infrastructure for a broken pipeline. A run with any skips is marked `passed: false` even if every extracted metric cleared its threshold.

### New CLI flags

- `--strict` — opt out of refusal-aware scoring, revert to v0.1.x semantics
- `--refusal-aware` — explicitly enable refusal-aware scoring (default behavior)
- `--no-refusal-aware` — alias for `--strict`

### New optional extra

```bash
pip install rag-forge-evaluator[ragas-voyage]
```

Installs `voyageai>=0.2` so `--judge claude --evaluator ragas` can use Voyage embeddings instead of requiring an `OPENAI_API_KEY`. Claude-native shops can now run RAGAS end-to-end without touching OpenAI.

## Bug fixes

- **`OpenAIEmbeddings.embed_query` AttributeError** on ragas 0.4.x (Cycle 2 Finding #4). Fixed by injecting `RagForgeRagasEmbeddings` so ragas never reaches for its internal LangChain-style defaults.
- **`gpt-4o-mini` max_tokens overflow** on long structured clinical responses (Finding #5). Fixed by routing through our own Judge wrapper — `ClaudeJudge`'s 4096-token default applies (raisable via `RAG_FORGE_JUDGE_MAX_TOKENS` env var), instead of ragas's hardcoded internal `gpt-4o-mini` limit.
- **RAGAS exceptions silently coerced to score 0.0** (Finding #6). Fixed by replacing the 0.0 fallback in `_extract_ragas_score` with a `ValueError` that the caller converts into a `SkipRecord`. Skips are visible in the report.
- **Out-of-range combined judge scores** (-1, 2.5, NaN, inf) were previously accepted as real metric values. Now rejected as malformed output and marked skipped.
- **Judge alias allowlist on the ragas path** rejected `claude-sonnet` even though `_create_judge()` accepted it. Fixed — ragas validation now reuses `_KNOWN_JUDGE_ALIASES` as the single source of truth.
- **MetricEvaluator subclass routing** — the combined single-call path's type check used `isinstance()`, so subclasses that overrode `evaluate_sample()` were silently rerouted into the generic path. Now uses exact-type set equality so subclasses correctly fall through to the per-metric path.
- **Compliance footer false claim** — previous copy claimed samples stayed "in-memory only" when in fact HTML/JSON/PDF artifacts persist them. Corrected.

## Migration guide

1. **Upgrade the Python evaluator:**
   ```bash
   pip install -U rag-forge-evaluator[ragas]
   ```
   ragas will upgrade to a supported version in the `>=0.4,<0.6` range automatically.

2. **If you use Claude judge with ragas**, also install Voyage embeddings:
   ```bash
   pip install rag-forge-evaluator[ragas-voyage]
   ```

3. **If your CI gates were tuned against v0.1.x scores**, re-run your baseline audits on v0.2.0 to establish new thresholds — scores will shift on pipelines with safety refusals.

4. **To preserve v0.1.x scoring semantics** exactly, add `--strict` to your audit commands.
