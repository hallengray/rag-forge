# RAG-Forge v0.2.0 — Evaluator Refresh Design

**Status:** Approved
**Author:** RAG-Forge (Femi Adedayo)
**Date:** 2026-04-14
**Supersedes:** v0.1.3

---

## Why this release exists

The PearMedica Cycle 2 audit (2026-04-14) ran RAG-Forge 0.1.3 against a real clinical RAG pipeline and exposed four structural problems with the evaluator:

1. **RAGAS integration is version-fragile.** The adapter relies on `ragas.evaluate()` picking its own default LLM and embeddings, which change between ragas major versions. On ragas 0.4.x this crashes with `AttributeError: 'OpenAIEmbeddings' object has no attribute 'embed_query'` and truncates long structured responses with `finish_reason='length'`.
2. **Exceptions are silently coerced to score 0.0.** A broken ragas run reports `Overall: 0.0000, Scored: 48, Skipped: 0`. The user believes their RAG is broken when the framework is broken.
3. **Safety refusals are penalized as quality failures.** When a pipeline correctly refuses to answer (because the knowledge base lacks sufficient support), the evaluator scores it as a non-answer. This misleads every safety-conscious RAG in every regulated industry.
4. **The audit report is unprofessional.** Text is cut off, HTML bleeds off-screen, per-sample detail is inaccessible in the PDF, alignment is inconsistent. As the single most customer-facing artifact RAG-Forge produces, this undermines the entire tool.

v0.2.0 addresses all four. The unifying theme: **the evaluator now understands what the pipeline was built to do, and communicates its findings with the seriousness the domain demands.**

This is a minor-version bump, not a patch, because the combined scope is feature-sized: a new scoring mode, a new embeddings provider, a rewritten report, and behavior changes that affect existing users' scores.

---

## Goals

- Fix the three RAGAS bugs (Findings #4, #5, #6 from the PearMedica Cycle 2 audit notes) at the **architectural root** — inject our own LLM and embeddings wrappers so ragas never reaches for its version-fragile defaults.
- Add **refusal-aware scoring** as a default-on behavior so correct safety refusals are no longer misclassified as quality failures, with a `--strict` escape hatch for users who want the old semantics.
- Rebuild the audit report to **top-tier professional standard**: responsive HTML, print-quality PDF, a shared template for both, comprehensive information architecture, and a compliance-grade footer.
- Ship all four changes as `v0.2.0` across the rag-forge-core, rag-forge-evaluator, rag-forge-observability, `@rag-forge/cli`, and `@rag-forge/mcp` packages.

## Non-goals

- **No new metrics.** Hand-rolled recall@k and nDCG@10 are deferred to v0.2.1.
- **No new evaluators.** DeepEval engine is untouched.
- **No core pipeline changes.** `packages/core` only gets a version bump for release consistency.
- **No offline / air-gapped embeddings.** `sentence-transformers` support is deferred to v0.2.1.
- **No vercel.ts migration** or other infrastructure churn unrelated to the evaluator.
- **No ragas 0.2.x compatibility.** Hard cut to `ragas>=0.4,<0.6` — users with a 0.2.x pin upgrade or stay on 0.1.3.

## Workstream A — RAGAS adapter rebuild (inject-and-own)

### Problem restated

Today's `packages/evaluator/src/rag_forge_evaluator/engines/ragas_evaluator.py` calls:

```python
result = ragas_evaluate(dataset, metrics=[...])
```

No `llm=`. No `embeddings=`. Ragas picks its own defaults, which changed in 0.4.x to a new `BaseRagasLLM` / `BaseRagasEmbeddings` interface. Our adapter is coupled to whichever default picker ragas ships today, which means every ragas upgrade is an implicit breaking change for us.

### Solution

A new module `packages/evaluator/src/rag_forge_evaluator/engines/ragas_adapters.py` exposes two wrapper classes that implement ragas's public `BaseRagasLLM` and `BaseRagasEmbeddings` interfaces. `ragas_evaluator.py` is refactored to build these wrappers from the existing `Judge` abstraction we already ship (Claude, OpenAI, Mock) and inject them explicitly.

```
                    ┌───────────────────────┐
                    │ audit.py              │
                    └──────┬────────────────┘
                           │ creates Judge
                           ▼
                    ┌───────────────────────┐
                    │ Judge (existing)      │
                    │ Claude/OpenAI/Mock    │
                    └──────┬────────────────┘
                           │ wrapped by
                           ▼
                    ┌───────────────────────┐
                    │ engines/ragas_adapters│  ← NEW
                    │  RagForgeRagasLLM     │
                    │  RagForgeRagasEmbed...│
                    └──────┬────────────────┘
                           │ implements
                           ▼
                    ┌───────────────────────┐
                    │ BaseRagasLLM          │
                    │ BaseRagasEmbeddings   │
                    └──────┬────────────────┘
                           │ consumed by
                           ▼
                    ┌───────────────────────┐
                    │ ragas_evaluate(       │
                    │   llm=..., embed=...) │
                    └───────────────────────┘
```

### Components

**`engines/ragas_adapters.py` (new, ~250 LOC)**

- `RagForgeRagasLLM(BaseRagasLLM)`:
  - Wraps a `Judge` instance plus a configurable `max_tokens` (default **8192**, raisable via `AuditConfig.ragas_max_tokens`).
  - Implements both `generate_text` (sync) and `agenerate_text` (async) — ragas calls both code paths depending on the metric.
  - Forwards prompts through the judge's existing `.complete()` method so overload-retry, Claude-specific headers, and progress reporting all continue to work.
  - Reports the judge's real model name via `model_name()`.
  - Deferred ragas imports (matches the existing `ragas_evaluator.py` pattern — module is importable without ragas installed).

- `RagForgeRagasEmbeddings(BaseRagasEmbeddings)`:
  - Constructor takes `provider: Literal["openai", "voyage"]` and an optional explicit model override.
  - `provider="openai"` uses `openai.embeddings.create` with `text-embedding-3-small` (default).
  - `provider="voyage"` uses `voyageai.Client().embed` with `voyage-3` (default).
  - Implements `embed_query` (single string) and `embed_documents` (batch).
  - Auto-selection rule: `ClaudeJudge` → `voyage`; `OpenAIJudge` → `openai`; `MockJudge` → deterministic stub (returns a hashed vector, no network call — used by the integration test).
  - Explicit override via `AuditConfig.ragas_embeddings_provider`.

**`engines/ragas_evaluator.py` (modified, ~40 LOC changed)**

- `__init__` now accepts `judge: Judge` (threaded from `create_evaluator` factory, same way llm-judge receives it).
- `evaluate()` constructs the two adapters from `self._judge` and passes them as `llm=` / `embeddings=` to `ragas_evaluate(...)`.
- The `_extract_ragas_score` helper's 0.0 fallback path is **deleted**. Exceptions during extraction now append a `SkipRecord` to `self._skipped_samples` and are not counted in the metric average.
- New instance field `self._skipped_samples: list[SkipRecord]` propagated into the returned `EvaluationResult`.

**`engine.py` (modified — shared interfaces)**

- `EvaluationResult` gains an optional `skipped_samples: list[SkipRecord]` field (default `[]`). Backward-compatible — llm-judge engine emits an empty list.
- `EvaluationResult` also gains optional `scoring_modes_count: dict[str, int]` — a count of samples per mode (`{"standard": 11, "safety_refusal": 1}`). Used by Workstream B.
- New dataclass `SkipRecord(sample_id: str, metric_name: str, reason: str, exception_type: str)`.

**`engines/__init__.py` / `create_evaluator` factory (modified)**

- Thread the `judge` instance into `RagasEvaluator(...)`, mirroring the existing llm-judge wiring.

**`audit.py` (modified)**

- Delete the hard block at lines 148–165 that refuses `--judge claude --evaluator ragas`. Replace with a capability check: if `--judge claude` is requested and `voyageai` is not installed and the embeddings provider is auto-selecting, raise `ConfigurationError` with the install hint `pip install rag-forge-evaluator[ragas-voyage]`.
- The banner line that hardcodes `"gpt-4o-mini (RAGAS internal — judge override ignored)"` is updated to report the real judge model.
- Surface `skipped_samples` count into the progress reporter's "run complete" banner.
- `AuditConfig` gains `ragas_max_tokens: int = 8192` (threaded into `RagForgeRagasLLM`) and `ragas_embeddings_provider: Literal["openai", "voyage"] | None = None` (None = auto-select from judge type).

**`pyproject.toml` (modified)**

- `[ragas]` extra changes from `ragas>=0.2.10,<0.3` to **`ragas>=0.4,<0.6`** (hard cut per decision D1).
- New `[ragas-voyage]` extra: `voyageai>=0.2`.
- No changes to `openai` or `anthropic` pins — both stay as core dependencies.

### Error handling

- `voyageai` not installed + Claude judge + ragas evaluator → `ConfigurationError` at config time, before any judge calls run. Clear install hint.
- Ragas version detected as 0.2.x at runtime (monorepo users with upstream pins) → `ConfigurationError` at config time with upgrade hint.
- `VOYAGE_API_KEY` missing + Voyage provider → `ConfigurationError` at config time.
- Per-sample exception inside ragas → caught, `SkipRecord` appended, run continues, visible in report (skip-and-average per decision S1).
- Whole-batch ragas crash → propagates as today; existing partial-report path in `audit.py:255` catches it.

## Workstream B — Refusal-aware scoring

### Problem restated

A RAG pipeline built with a strict "refuse if unsupported" guardrail (PearMedica's paediatric dosing refusal, legal "I can't advise on jurisdiction X", finance "no source data for this instrument", etc.) is *correctly* refusing to fabricate. The evaluator, not knowing the pipeline's design intent, treats the refusal as a non-answer and assigns low answer-relevance and low context-relevance scores. **The safety rail is being punished.** This affects any safety-conscious RAG in any regulated industry, not just clinical.

### Solution

Default-on **refusal-aware scoring mode**. The same judge call that scores each sample now first classifies the sample as `standard` or `safety_refusal` (per decision R2 — judge-driven inline classification, no separate call), then branches on the classification. A `--strict` escape hatch reverts to v0.1.3 semantics. A prominent in-report banner warns when the refusal rate is unexpectedly high.

### Why default-on (plain English)

When v0.2.0 ships, every user who runs an audit gets refusal-aware scoring automatically. Their v0.1.3 scores and v0.2.0 scores will differ on the same pipeline — some cases that scored low before will now score high, not because the RAG improved but because the evaluator stopped penalizing correct refusals. This is the right behavior: the old scores were wrong in a systematic way, and opt-in would leave most users with wrong scores by default.

Three mitigations are required to make default-on safe:

1. **Prominent release-notes callout**: *"If your pipeline has safety refusals, your scores will go up. This is a correction, not a regression."*
2. **A `--strict` escape hatch** (`refusal_aware: false` in config) that reverts to v0.1.3 scoring semantics for users who deliberately want the strict penalty as a signal.
3. **A "spot-check" warning banner** at the top of every report when the refusal rate exceeds 30%, prompting the user to audit the judge's classifications in the per-sample detail.

### Classification prompt injection

Both the llm-judge path (`metrics/llm_judge.py`) and the RAGAS path (via the `RagForgeRagasLLM` wrapper) inject a classification preamble into the scoring prompt:

> *Before scoring, determine two things: (a) Does the retrieved context contain sufficient specific information to answer the question with concrete recommendations? (b) Does the response refuse or abstain from providing specific recommendations, citing insufficient knowledge-base support?*
>
> *If (a) = no AND (b) = yes, this is a **valid safety refusal**. Score it on the refusal rubric. Otherwise score on the standard rubric.*
>
> *Return your classification as `scoring_mode: "safety_refusal" | "standard"` with a one-line justification in `refusal_justification`.*

The judge returns both the scoring mode and the justification in its structured response.

### Refusal rubric

Applied only when `scoring_mode == "safety_refusal"`:

| Metric | Standard rubric | Refusal rubric |
|---|---|---|
| Answer relevance | Did the response answer the question? | Did the response acknowledge the question and explain *why* it cannot answer? |
| Faithfulness | Are the claims supported by context? | Did the response avoid making any claims not supported by context? (pure refusal → 1.0) |
| Context relevance | Are retrieved chunks directly relevant to the answer? | Are retrieved chunks topically related? (softer) |
| Hallucination | Were unsupported facts introduced? | Were unsupported facts introduced? (pure refusal → 1.0) |

### Components

**`metrics/llm_judge.py` (modified, ~120 LOC changed)**

- Inject the classification preamble into the scoring prompt.
- Parse the new judge response shape: `{scoring_mode, refusal_justification, faithfulness, answer_relevance, context_relevance, hallucination}`.
- Branch on `scoring_mode` when interpreting the scores.
- Populate the new `SampleScore.scoring_mode` and `SampleScore.refusal_justification` fields.

**`engines/ragas_adapters.py` (modified from Workstream A, ~30 LOC added)**

- `RagForgeRagasLLM.generate_text` transparently prepends the same classification preamble when called by ragas metrics. The judge returns both scoring mode and metric score in one response; the wrapper parses and threads the mode through to `RagasEvaluator._skipped_samples` / `_scoring_modes`.

**`engine.py` (modified, additive)**

- `MetricResult` gains optional `scoring_mode: Literal["standard", "safety_refusal"] | None` and `refusal_justification: str | None`.
- `EvaluationResult` gains `scoring_modes_count: dict[str, int]` for aggregate reporting.
- `SampleScore` gains the same two fields.

**`audit.py` / `AuditConfig` (modified)**

- `AuditConfig` gains `refusal_aware: bool = True` (default-on).
- New CLI flag `--strict` (boolean, default false) sets `refusal_aware=False`. An equivalent `--refusal-aware` / `--no-refusal-aware` pair exists for explicitness.
- Progress reporter banner shows `Refusal-aware: ON / OFF` at run start.

**New test file `tests/test_refusal_aware_scoring.py` (~15 cases)**

- Mock judge returning both `standard` and `safety_refusal` classifications with structured justifications.
- Covers: standard scoring unchanged; refusal rubric applied correctly; `--strict` reverts to standard-only; refusal justification propagates to `EvaluationResult`; aggregate `scoring_modes_count` accurate.

### Risk: judge mis-classification

The judge can be wrong. If Claude misclassifies an evasion as a refusal, users get silently inflated scores. Mitigations:

- Every report prominently shows `safety_refusal_rate` in the run manifest strip and inside the new Safety Refusals block.
- When the refusal rate exceeds **30%**, the report renders an amber banner: *"High refusal rate detected — spot-check the judge's classifications in the per-sample detail."*
- The judge justification is stored per-sample in the report so operators can audit.
- Documentation explicitly states: *"This mode trusts the judge's classification. Spot-check at least one sample per mode on your first run."*

## Workstream C — Report rewrite

### Problem restated

From the Cycle 2 feedback:

> *"text are cut off, not well aligned, cant expand the per-sample breakdown in pdf version and the html version is cut off and bleeding out of the screen. This is not professional."*

The audit report is the single most customer-facing artifact RAG-Forge produces. Regulated-industry users will attach it to compliance submissions; teams paste it into PRs; executives read it in Slack previews. It must meet top-tier standard.

### Solution

A single HTML template with a print stylesheet, rendered via Playwright for PDF generation. Fix once, both formats benefit. The visual design is serif / formal / "Bloomberg-or-Deloitte" style (Option C from the brainstorming session), which matches the seriousness regulated-industry users expect.

### Information architecture

The first page of the report follows this order:

1. **Header strip** — RAG-Forge wordmark (left), Report № + issue date + cycle + UTC timestamp (right).
2. **Title block** — Project name in large serif italic, divider, one-line project description.
3. **Run manifest strip** — 6-cell horizontal bar: Evaluator · Judge Model · Samples (+ skipped count) · Wall Time · Cost · RAG-Forge version.
4. **RMM hero callout** — gold-accented block containing:
   - Level name + plain-English "what this level means and what's needed for the next"
   - Historical sparkline (SVG) showing score trend across prior cycles with delta vs Cycle 1
   - Overall score in large serif italic with "N of 4 passing" subtext
5. **The maturity ladder** — horizontal 6-column strip showing Levels 0–5 with current position highlighted in gold and cleared levels marked in green.
6. **Executive summary** — prose paragraph explaining what the audit measured and what it found.
7. **TL;DR · Action Required** — gold-bordered callout with three parts:
   - Left column: "✓ What's working" (3 bullets)
   - Right column: "✗ What needs fixing" (3 bullets)
   - Footer band: "→ Priority next step" (one specific actionable recommendation)
8. **Metric breakdown table** — 5 columns: Metric · What it measures (plain English) · Score · Target · Status.
9. **Cost summary block** — Total spend (large serif italic) · Per-sample average · Input tokens + split · Output tokens + split, with a footer projecting 100-sample and monthly-CI costs.
10. **Safety refusals block** (green-accented) — refusal rate, plain-English explanation, per-refused-case detail cards with judge justification.
11. **Worst performing case callout** — sample ID, headline metrics, one-sentence diagnosis, pointer to per-sample detail page.
12. **Per-sample detail section** — one card per sample: query, response, retrieved context, per-metric scores, scoring mode + justification, wall time. In HTML these use `<details>/<summary>` for collapse; in print mode every card is fully expanded with `page-break-inside: avoid`.
13. **Skipped samples section** (only rendered if non-empty) — per-sample reason and exception type.
14. **Compliance footer** — double-rule divider, then:
    - Section header: "Compliance & Attestation"
    - Two-column: Method Disclosure · Data Handling
    - Regulatory Alignment block (soft-gray background) referencing NIST AI RMF 1.0 § 4.3, ISO/IEC 42001:2023 § 8.3, EU AI Act Article 15
    - Three-column attestation grid: Authored By (`RAG-Forge` in Georgia italic) · Issued (date + UTC) · Valid Until (next material change or 90 days)
    - Limitations & disclaimer paragraph
    - Footer slug: product name, GitHub URL, Report № · Page N of M

### Technical approach

**Single source of truth.** One Jinja template, one embedded CSS block, two rendering modes:

- **Browser mode** — interactive. `<details>/<summary>` collapse for per-sample cards. Responsive CSS Grid with max-widths and breakpoints. No horizontal bleed at any viewport down to mobile.
- **Print mode** — `@media print` stylesheet plus Playwright's `emulateMedia({media: 'print'})`. All `<details>` elements are forced open. `page-break-inside: avoid` on cards. `@page` rules set margins, page numbers (`@page { @bottom-right { content: counter(page) " / " counter(pages); } }`), and header/footer bands.

**Typography and whitespace.**

- Serif display: `Georgia, "Times New Roman", serif` for titles, overall score, RMM level name, authored-by, cost total.
- Sans system stack: `system-ui, -apple-system, "Segoe UI", sans-serif` for body, tables, metric labels, manifest cells.
- Monospace: `ui-monospace, "SF Mono", Menlo, monospace` for sample IDs and package versions.
- Color palette: neutral grays (#111, #333, #666, #999, #d4d4d4, #e5e5e5, #fafafa, #fff) + semantic accents (pass #1a5f1a / #16a34a, fail #9a1111 / #dc2626, highlight gold #9a7b2e).
- Generous whitespace, 1.7 line-height on body prose, `font-variant-numeric: tabular-nums` on all score columns.

**No truncation anywhere.** `overflow-wrap: anywhere` + `hyphens: auto` on long text. Tables wrap in `overflow-x: auto` containers in HTML; in print mode they break across pages cleanly with table header repetition.

**Historical sparkline.** SVG generated inline from the `history.py` data already captured by RAG-Forge. Shows up to 8 most recent runs. Current run is emphasized with a filled circle; delta vs first point is labeled underneath.

### Components

**`report/templates/audit.html.j2` (new, ~400 LOC template + 600 LOC embedded CSS)**

The main Jinja template. Receives a fully-hydrated context dict from `generator.py`. All sections 1–14 above are implemented as Jinja macros for maintainability.

**`report/generator.py` (rewritten, ~300 LOC)**

- `generate_html(report: AuditReport) -> str` builds the context dict and renders the template.
- Context includes: header fields, manifest cells, score + sparkline data points, ladder state, exec summary, TL;DR buckets (auto-generated from threshold deltas), metric rows, cost breakdown, refusal data, worst-case, per-sample list, skipped list, compliance block.
- TL;DR "priority next step" is generated deterministically: picks the worst-performing metric, finds the sample with the largest gap, produces a one-sentence recommendation.

**`report/pdf.py` (modified, ~40 LOC changed)**

- Launches Playwright, loads the same HTML via a local file URL.
- Calls `page.emulate_media({"media": "print"})` before `page.pdf(...)`.
- `page.pdf` options: `format="A4"`, `margin={top:"20mm",right:"18mm",bottom:"22mm",left:"18mm"}`, `print_background=True`, `display_header_footer=False` (the template's own `@page` rules handle this).

**`report/json_report.py` / `report/__init__.py` (modified, additive only)**

- JSON report gains top-level `skipped_samples`, `scoring_modes_count`, `safety_refusal_rate`, and per-sample `scoring_mode` + `refusal_justification`.
- Existing JSON fields are unchanged. Additive only — downstream consumers don't break.

**Visual regression test (new)**

- `tests/test_report_visual_regression.py` uses Playwright to render a golden-fixture audit report to PNG at 1440×900 and compares pixel-by-pixel to a committed baseline.
- Runs opt-in via `pytest -m visual` because Playwright screenshots are OS-dependent; the baseline is captured on the CI runner and re-captured when the template intentionally changes.

### What we remove

- The current `report/generator.py` layout logic is replaced wholesale.
- Any hardcoded width/height values in `report/pdf.py` are removed in favor of print CSS.
- The current `is_available()` gate in `report/pdf.py` stays; dependency check on Playwright is unchanged.

## Testing strategy

Gated in layers:

**Layer 1 — unit tests, no ragas, no network.**

- `tests/test_ragas_adapters.py` (new, ~20 cases) — mock Judge + mock voyageai/openai clients. Verifies: sync/async generate_text; embeddings dispatch; batch vs single; provider auto-selection from judge type; `--strict` path.
- `tests/test_refusal_aware_scoring.py` (new, ~15 cases) — mock judge returning both classifications; verifies rubric branching, justification propagation, aggregate counts.
- `tests/test_ragas_extractor.py` (extended) — exceptions now produce `SkipRecord`s, never 0.0. Overall score averaged over successful samples. Empty results → `passed=False`, `samples_evaluated=0`.
- `tests/test_report_generator.py` (new) — HTML rendering is deterministic for a golden fixture; all template variables are populated; no missing Jinja keys.

**Layer 2 — integration, real ragas 0.4.x, mock judge.**

- `tests/test_ragas_integration.py` (new, gated by `pytest -m ragas_integration`) — real ragas import, real BaseRagasLLM/Embeddings interface, `MockJudge` returning deterministic JSON, 2-sample tiny fixture, one full `evaluate()` round-trip. Asserts non-zero scores. This is the **contract test** — if ragas 0.4.2 breaks our interface, this goes red.
- Regression fixture: a 3-sample sanitized excerpt of PearMedica's `run-2026-04-14.jsonl`. Runs with mock judge. Asserts the v0.1.3 failure signatures (`AttributeError`, `finish_reason='length'`, `0.0 across the board`) are gone.

**Layer 3 — visual regression.**

- `tests/test_report_visual_regression.py` (new, gated by `pytest -m visual`) — Playwright screenshot comparison of the rendered report against a committed PNG baseline at 1440×900.

**Layer 4 — end-to-end smoke.**

- A `just smoke` / `pnpm smoke` recipe that runs `rag-forge-eval audit` against a 2-sample fixture with mock judge under both `--evaluator llm-judge` and `--evaluator ragas`, asserts exit code 0 and report artifacts exist.

## Decisions locked during brainstorming

1. **Option D** — inject our own LLM + embeddings wrappers. Not patch-per-symptom, not version-pinning, not dual-support.
2. **D1** — hard cut on ragas: `>=0.4,<0.6`. No dual-code-path for 0.2.x.
3. **E2** — OpenAI + Voyage embeddings. `sentence-transformers` / offline mode deferred to v0.2.1.
4. **S1** — skip-and-average on ragas exceptions, matching llm-judge semantics.
5. **Refusal-aware scoring included in this release** — same structural insight as Findings #4/#5/#6.
6. **Default-on** with `--strict` escape hatch and prominent refusal-rate banner.
7. **R2** — judge-driven inline classification, no separate API call.
8. **V2** — ship as v0.2.0, not v0.1.4. Combined scope and breaking-change surface exceeds patch semantics.
9. **Report design direction: Option C** (Bloomberg / Deloitte serif). Detailed with RMM hero, full ladder, sparkline, run manifest, cost summary, TL;DR, compliance footer (no fingerprint). Author is `RAG-Forge`.
10. **TL;DR placement** below score + ladder + executive summary — the TL;DR synthesizes the numbers and prose above it.

## Out of scope (explicit)

- Hand-rolled recall@k / nDCG@10 script → v0.2.1.
- Offline / air-gapped embeddings via sentence-transformers → v0.2.1.
- DeepEval engine improvements → future.
- Report fingerprint / SHA-256 attestation → removed per user request.
- Refusal-aware default-on across custom user evaluators → users who subclass `EvaluatorInterface` must opt in manually.
- Changes to `packages/core` beyond version bump.
- Changes to CLI telemetry or observability instrumentation beyond version bump.

None. All design decisions are locked.
