## Cycle 2 Finding #4 — `OpenAIEmbeddings.embed_query` AttributeError

**Verdict: VERIFIED FIXED.**

### What Cycle 2 saw

Cycle 2 (2026-04-14, v0.1.3) observed this exception 2x in the RAGAS run against the PearMedica golden set:

```text
AttributeError: 'OpenAIEmbeddings' object has no attribute 'embed_query'
Exception raised in Job[41], Job[44], Job[45]
```

Root cause was that ragas 0.4.x's `OpenAIEmbeddings` shape dropped the `embed_query` method in favor of a new `Embeddings` interface, and `rag-forge-evaluator/engines/ragas_evaluator.py` was still calling `.embed_query()` on it.

### What Cycle 3 could not verify

Cycle 3 (2026-04-15, v0.2.1) could not verify the fix because v0.2.1's entirely separate C3-2 bug (`RagForgeRagasLLM.generate` missing) crashed the engine before any embeddings code was reached. The RAGAS pass produced 0/48 scored samples with 6 seconds wall time — no network round-trip, no embeddings call.

### What v0.2.2 verifies

Two layers of evidence:

1. **Contract test** (`tests/test_ragas_adapters_contract.py::test_embeddings_wrapper_declares_every_base_class_public_method`) iterates every public method on `ragas.embeddings.base.BaseRagasEmbeddings` and asserts `RagForgeRagasEmbeddings` declares a callable of the same name. Methods enumerated at run time on the installed ragas 0.4.3:

   - `aembed_documents`
   - `aembed_query`
   - `embed_documents`
   - `embed_query`
   - `embed_text`
   - `embed_texts`
   - `set_run_config`

   All seven exist on our wrapper after v0.2.2's G1 changes.
2. **End-to-end smoke** (`tests/test_ragas_adapters_e2e.py::test_ragas_evaluate_never_raises_attribute_error_on_wrapper`) runs `ragas.evaluate()` on a 1-sample dataset passing `RagForgeRagasEmbeddings(provider="mock")`, then checks that no `AttributeError` referencing our wrapper classes was raised. The test completes with the stub judge without hitting any embeddings-related AttributeError.

## Cycle 2 Finding #5 — `max_tokens` overflow on long structured clinical responses

**Verdict: VERIFIED FIXED.**

### What Cycle 2 saw

Cycle 2 observed 4x `finish_reason='length'` and 3x `InstructorRetryException` during the RAGAS faithfulness statement-extraction pass:

```text
ChatCompletion(... finish_reason='length', ...
  usage=CompletionUsage(completion_tokens=3072 / 2048 / 1024 ... ))
InstructorRetryException: The output is incomplete due to a max_tokens length limit.
```

Root cause was that PearMedica's structured clinical responses (`conditions`, `triage`, `differential_diagnoses`, `next_question`, `red_flags`, `treatment_guidance`) produce 20-50 extractable statements per case, and `gpt-4o-mini`'s default `max_tokens` truncated the judge's statement-extraction output mid-response.

### What v0.2.0 changed

v0.2.0 raised `ClaudeJudge` default `max_tokens` from 1024 to 4096. The adapter now also honors `AuditConfig.ragas_max_tokens` (default 8192). Source: the existing v0.2.0 release notes.

### What Cycle 3 could not verify

Same masking problem as Finding #4: v0.2.1's C3-2 crash prevented RAGAS from reaching the statement-extraction step at all.

### What v0.2.2 verifies

The v0.2.2 test suite includes a dedicated long-structured-response regression test that reads the Cycle 2 fixture (`tests/fixtures/ragas_regression/cycle-2-excerpt.jsonl`, 3 samples from the real audit with responses sanitized to remove PHI) and asserts:

1. The response is long enough to trigger Finding #5 (`> 50 words per response, producing 20+ extractable statements`).
2. No skip record in the result has an `exception_type` of `"InstructorRetryException"`.
3. No skip record has a `reason` containing `"finish_reason='length'"`.

## Cycle 2 Finding #6 — silent 0.0 coercion

**Verdict: PARTIALLY FIXED in v0.2.0, FULLY FIXED in v0.2.2.**

### What Cycle 2 saw

Cycle 2 reported `"Scored: 48, Overall: 0.0000"` — every metric at 0.0 with `Skipped: 0`. v0.1.3 caught RAGAS exceptions at a low level in `ragas_evaluator.py` and silently coerced them into 0.0 "scored" entries, double-counting broken infrastructure as "everything scored zero."

### What v0.2.0 fixed

v0.2.0 replaced the 0.0 fallback with `SkipRecord` tracking. The `skipped_samples` list on `EvaluationResult` is now populated, and skip records surface in both the JSON and HTML/PDF reports.

### What Cycle 3 still saw

Cycle 3 reported `"But Skipped: 0 is still wrong — the 48 exceptions should flow into the skipped counter, not disappear."` Two residual gaps:

1. The `skipped_evaluations` integer counter on `EvaluationResult` was never populated by the RAGAS engine — only the `skipped_samples` detail list. The report's top-level "Skipped: N" line reads from the integer counter and reported `0` even when every job had crashed.
2. Individual-metric extraction failures produced one aggregate `SkipRecord` per metric name (`sample_id="<aggregate>"`) instead of fanning out per (sample, metric) pair. A 12-sample x 4-metric run that failed entirely produced 4 skip records instead of 48.

### What v0.2.2 fixes

Workstream G3:

1. `skipped_evaluations` is now set to `len(skipped_samples)` on both return sites in `RagasEvaluator.evaluate()`. A new regression test (`test_skipped_evaluations_counter_matches_skip_record_length`) asserts the two can never drift apart silently again.
2. Skip-record creation is extracted into `_fan_out_skip_records()` which produces one record per (sample, metric) pair with the real sample_id attached. A new regression test (`test_whole_batch_crash_fans_out_to_every_sample_metric_pair`) asserts a 3-sample x 4-metric run produces exactly 12 skip records with real sample IDs.
3. Reason strings are truncated to 400 chars with trailing ellipsis (`test_skip_reason_truncated_to_400_chars`).

## Cycle 3 Finding C3-2 — `RagForgeRagasLLM.generate` missing

**Verdict: VERIFIED FIXED.**

### What Cycle 3 saw

```text
Exception raised in Job[0]: AttributeError('RagForgeRagasLLM' object has no attribute 'generate')
Exception raised in Job[1]: AttributeError('RagForgeRagasLLM' object has no attribute 'generate')
... (repeats identically for Job[0] through Job[47], all 48 jobs)
Evaluating: 100%|##########| 48/48 [00:00<00:00, 1402.31it/s]
```

6-second wall time, 0 network round-trips, 0/48 scored. The generate method wasn't on our duck-typed wrapper because the adapter's module docstring incorrectly claimed `"ragas only calls generate_text / agenerate_text / model_name"`.

### What v0.2.2 fixes

v0.2.2's G1 workstream adds duck-typed shims for every public method on `ragas.llms.base.BaseRagasLLM`:

- `async def generate(prompt, n, temperature, stop, callbacks) -> LLMResult`
- `def is_finished(response) -> bool`
- `def get_temperature(n) -> float`
- `def set_run_config(run_config) -> None`
- `run_config: Any = None` attribute

Plus the contract test that asserts every public method on the real `BaseRagasLLM` is declared on our wrapper, so the next ragas release that grows a new method fails in CI instead of in a user audit.

## What would still falsify this report

This report is an artifact of the rag-forge repository's own test suite against sanitized fixtures. It proves the v0.2.2 code no longer crashes on any of the signatures Cycles 2 and 3 documented. It does **not** prove the v0.2.2 code would produce higher RAGAS scores on the real PearMedica Cycle 4 run — that requires running the evaluator against live telemetry with a real judge, which is outside the scope of this release's tests.

If a Cycle 4 run against the PearMedica `eval/telemetry/run-2026-04-15.jsonl` fixture under v0.2.2 + a real `claude-sonnet-4-6` judge + Voyage embeddings shows any of these signatures return:

- `AttributeError` mentioning `RagForgeRagasLLM`, `RagForgeRagasEmbeddings`, `OpenAIEmbeddings`, or `embed_query`
- `InstructorRetryException` or `finish_reason='length'`
- `skipped_evaluations: 0` while every score is 0.0
- A metric with `sample_id="<aggregate>"` in any skip record

then this report is falsified and v0.2.2 did not do its job. The release-ready bar is "Cycle 4 re-runs the same golden set under v0.2.2 and produces non-empty RAGAS metrics or skip records with real error signatures, not empty both."
