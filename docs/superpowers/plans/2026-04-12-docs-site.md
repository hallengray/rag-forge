```jsx
import { generateStaticParamsFor, importPage } from "nextra/pages";
import { useMDXComponents as getMDXComponents } from "../../mdx-components";

export const generateStaticParams = generateStaticParamsFor("mdxPath");

export async function generateMetadata(props) {
  const params = await props.params;
  const { metadata } = await importPage(params.mdxPath);
  return metadata;
}

const Wrapper = getMDXComponents().wrapper;

export default async function Page(props) {
  const params = await props.params;
  const result = await importPage(params.mdxPath);
  const { default: MDXContent, toc, metadata, sourceCode } = result;
  return (
    <Wrapper toc={toc} metadata={metadata} sourceCode={sourceCode}>
      <MDXContent {...props} params={params} />
    </Wrapper>
  );
}
```

---

# Core Concepts

> The mental model before you build.

## Pipeline stages

RAG-Forge treats a RAG pipeline as a sequence of stages. Each stage has its own CLI command, its own configuration, and its own observability.

1. **Ingest** (`add`) — pull source documents into the project
2. **Parse** (`parse`) — extract text, tables, and structure
3. **Chunk** (`chunk`) — split parsed content into retrieval units
4. **Index** (`index`) — embed chunks and write to a vector store
5. **Query** (`query`) — retrieve and generate an answer
6. **Evaluate** (`audit`, `assess`) — measure retrieval and answer quality

## Configuration

Every project has a `rag-forge.config.ts` at its root. This is the source of truth for chunking, retrieval, evaluation thresholds, and provider choices. RAG-Forge validates it at init time — misconfigurations fail fast.

## The RAG Maturity Model

The RMM is a 0-to-5 scale for RAG pipelines. RAG-Forge measures your pipeline against the RMM on every `audit` run, so "improving RAG" becomes a concrete, measurable goal. See [RAG Maturity Model](/concepts/rmm) for the full ladder.

## Observability by default

Every pipeline stage emits an OpenTelemetry span. Plug in Langfuse or any OTEL-compatible backend and you get request-level tracing with zero extra code. See [Observability](/concepts/observability).
```

> Every `rag-forge` command at a glance.

RAG-Forge's CLI is organized around pipeline stages. Most projects need only five commands: `init`, `add`, `index`, `query`, `audit`. The rest are there when you need them.

## Project lifecycle

| Command | Purpose |
|---------|---------|
| [`init`](/cli/init) | Scaffold a new RAG-Forge project from a template |
| [`add`](/cli/add) | Register documents with the project |
| [`parse`](/cli/parse) | Extract text from source files |
| [`chunk`](/cli/chunk) | Split parsed content into retrieval units |
| [`index`](/cli/index-command) | Embed chunks and write to the vector store |
| [`query`](/cli/query) | Run a retrieval + generation query |
| [`inspect`](/cli/inspect) | Inspect indexed chunks and metadata |

## Evaluation and quality

| Command | Purpose |
|---------|---------|
| [`audit`](/cli/audit) | Run the golden-set evaluation and score against RMM |
| [`assess`](/cli/assess) | One-shot assessment of an existing pipeline |
| [`golden`](/cli/golden) | Manage the golden question/answer set |
| [`drift`](/cli/drift) | Detect retrieval drift against a baseline |
| [`report`](/cli/report) | Generate a pipeline health HTML report |

## Operations

| Command | Purpose |
|---------|---------|
| [`cost`](/cli/cost) | Track LLM and embedding spend |
| [`cache`](/cli/cache) | Inspect the semantic cache |
| [`guardrails`](/cli/guardrails) | Run guardrail tests and PII scans |
| [`serve`](/cli/serve) | Run the MCP server (`--mcp`) |
| [`n8n`](/cli/n8n) | Export a project as an n8n workflow |
```

# MCP Server

> Expose RAG-Forge to Claude Desktop, Cursor, and other Model Context Protocol clients.

## What you get

The RAG-Forge MCP server wraps your indexed pipeline as a set of tools any MCP client can call. Out of the box, clients get:

- **Search** — run a retrieval query against your vector store
- **Inspect chunks** — pull raw chunks by id or metadata filter
- **Run evaluation** — trigger an audit against the golden set

## Running the server

```bash
rag-forge serve --mcp
```

By default this starts the server on `stdio` for Claude Desktop. For HTTP (SSE) transport on port 3333:

```bash
rag-forge serve --mcp --http --port 3333
```

The HTTP server supports multiple concurrent MCP clients — each connection gets its own server instance and session id. See `packages/mcp/src/transports/http.ts` for the implementation.

## Claude Desktop configuration

Add to `claude_desktop_config.json`:

```json
{
  "mcpServers": {
    "rag-forge": {
      "command": "rag-forge",
      "args": ["serve", "--mcp"]
    }
  }
}
```

## Transports

| Transport | When to use |
|-----------|-------------|
| `stdio` | Local desktop clients (Claude Desktop, Cursor) |
| `HTTP (SSE)` | Multi-client or networked deployments |

## Related

- [`rag-forge serve`](/cli/serve) — the command that starts the server
```
